# PNG-тайлсет: шаблон стыков и импорт

Цель: уйти от процедурного пиксель-арта к полноценным PNG-тайлам,
которые владелец рисует/генерирует во внешнем инструменте.
Скоуп первой итерации: **террейн + реки** (здания/юниты — позже,
фолбэк на процедурную графику встроен). Целевое разрешение — **×4**:
треугольник террейна 14×16 → **56×64** (согласовано 2026-07-05).

## Почему шаблон именно такой

Террейн рендерится по dual-сетке треугольников (◀▶) между центрами
гексов: у треугольника 3 угла, слои тайлов накладываются по порядку `T`
(вода → пляж → болото → луга → кустарник → лес → скалы → горы), стык —
это маска углов. Поэтому полный набор стыков для одного типа террейна:
2 ориентации × (6 угловых масок + 2 полных варианта) = 16 спрайтов.
Это dual-grid-автотайлинг в духе Tilesetter/Townscaper, только на
треугольниках — движку нужны лишь картинки с правильной альфой.

## Файлы

- `assets/tileset_template.png` — шаблон: в каждом слоте текущий спрайт
  ×4 как плейсхолдер (нейронке/художнику видно, что заменять и какой
  формы должна быть альфа). Фон прозрачный.
- `assets/tileset_markup.png` — оверлей той же геометрии: рамки слотов
  и имена (розовым). Наложить для ориентировки, УБРАТЬ перед экспортом.
- `assets/tileset_map.json` — разметка: имя спрайта → {x,y,w,h}.
  По ней работает и генерация, и импорт. Не менять руками геометрию,
  если не перегенерирован шаблон.
- `assets/tileset.png` — РЕЗУЛЬТАТ (кладёт владелец): лист той же
  геометрии с нарисованными тайлами.

## Правила рисования слотов

- Треугольник ◀/▶ занимает весь слот 56×64; пиксели вне треугольника
  оставлять прозрачными (движок кладёт тайлы встык по сетке).
- Переходные маски `tri<t>_<orr>_<bits>`: непрозрачной должна быть
  примерно та же зона, что у плейсхолдера (углы с битами=1), кромка —
  на вкус; полностью прозрачный слот = «не нарисовано» → игра возьмёт
  процедурный спрайт (можно сдавать лист по кусочку).
- Реки `rt_<orr>_<m>`: только русло с мягкими краями/берегами на
  ПРОЗРАЧНОМ фоне — один спрайт ложится на любой террейн
  (в отличие от старых процедурных 7 подкрасок берега).
- Полные варианты `full0/full1` — два разных шума одной фактуры,
  движок чередует их по хэшу клетки.

## Цикл работы

```
1. Кнопка «Шаблон тайлсета» в дебаг-панели (~) → скачаются
   tileset_template.png + tileset_markup.png + tileset_map.json
   (актуальные версии также закоммичены в assets/).
2. Генерация/рисовка листа поверх шаблона → сохранить как
   assets/tileset.png (геометрию слотов не двигать!).
3. node build.mjs   # увидишь «PNG-тайлсет: ВКЛ» — лист вшит в index.html
4. Открыть index.html → кнопка «Тайлсет-превью» (атлас с нарезкой),
   плюс обычная карта. Пустые слоты = процедурный фолбэк.
```

## Как устроен импорт (для следующих сессий)

- `build.mjs::tilesetInject` — инлайнит `assets/tileset.png` +
  `tileset_map.json` в бандл как `TILESET_PNG`/`TILESET_MAP`
  (base64 data-URI: однофайловость и никакого CORS-taint у canvas).
- `19_atlas.js`: `loadTileset(cb)` грузится один раз до первого
  `buildAtlas` (boot ждёт); слот считается занятым, если есть пиксели
  с альфой>10. `tsPlace(name)` кладёт слот в атлас (атлас растёт до
  512×2048 при активном тайлсете); иначе — процедурный painter.
  Река `rt_<orr>_<m>` регистрируется под всеми тинтами
  `rt_<tint>_<orr>_<m>`.
- Рендер не менялся: спрайт большего размера в атласе автоматически
  даёт больше текселей на тот же мировой квад (NearestFilter).

## Бэклог

- Шаблон второй очереди: дороги (композит из 6 сегментов + центр),
  здания/юниты/логова/фичи (простые именованные слоты).
- Превью швов: тестовая полоса «каждый слой на каждом подслое».
- Если ×4-атлас окажется тяжёл для телефона — параметр TS_SCALE=2.
