# Research: полноценный WFC для Fantasy Marka

Ветка: `research/full-wfc-oskar-islands`
Статус: research / decision gate. Код генератора пока не менялся.

Цель документа — зафиксировать исследование и рамку согласования перед проектированием полноценного WFC. Следующий шаг после согласования: отдельный `docs/wfc-design.md`, затем только после approval — реализация.

## 1. Что сейчас есть в проекте

Текущий генератор мира находится в `src/02c_worldgen.js`.

Пайплайн текущего мира:

1. `wgIsland()` — остров / море по краям / крупнейшая компонента суши.
2. `wgLakes()` — озёра среднего пояса.
3. `wgRidges()` — горные хребты-графы.
4. `wgElevation()` — поле высоты.
5. `genRivers()` — реки по dual-сетке / границам гексов.
6. `wgFill()` — WFC-like заполнение `GRASS / FOREST / ROCK`.
7. `wgEntities()` + `wgValidate()` — сущности и валидация мира.

Важно: текущий `wgFill()` — не полноценный WFC. Это стохастическая волна от фиксированных контуров с мягкими весами, соседской аффинностью и релаксацией. В нём нет доменов вариантов на клетку, минимальной энтропии, hard adjacency constraints, propagation, contradictions и backtracking. Это хороший procedural fill, но не constraint solver.

## 2. Что считать полноценным WFC

Для этой задачи под «полноценным WFC» предлагаю считать не просто визуальную волну, а constraint-solving слой со следующими обязательными частями:

- `wave`: для каждой клетки хранится domain возможных тайлов / паттернов.
- `tiles`: конечный набор тайлов с весами, тегами и правилами совместимости.
- `compat`: таблица допустимых соседств по 6 направлениям hex-сетки.
- `observe`: выбор не схлопнутой клетки с минимальной ненулевой энтропией.
- `collapse`: выбор одного варианта по весам, seed-deterministic.
- `propagate`: удаление несовместимых вариантов у соседей через очередь.
- `contradiction`: состояние, где domain пуст.
- `recovery`: bounded restart и/или backtracking.
- `constraints`: предзаданные клетки, маски зон, глобальные проверки связности.
- `trace`: debug replay не только порядка collapse, но и entropy/domain/contradictions.

Иначе мы снова получим WFC-like fill, а не WFC.

## 3. Research: базовые источники

### Maxim Gumin / WaveFunctionCollapse

Оригинальная популярная реализация WFC описывает output как wave, где каждый элемент содержит boolean coefficients допустимых паттернов. Алгоритм повторяет observation-propagation цикл: выбрать элемент с минимальной Shannon entropy, схлопнуть его в конкретный паттерн по весам, затем распространить последствия. Если у клетки все коэффициенты стали нулём — это contradiction.

Источник:
https://github.com/mxgmn/WaveFunctionCollapse

Ключевые выводы для проекта:

- минимальная энтропия нужна, чтобы не было сильного directional bias;
- propagation должен реально удалять варианты, а не просто менять веса;
- contradiction — нормальная часть WFC, её нельзя «замести» fallback-травой;
- WFC хорошо комбинируется с внешними constraints / ручным или процедурным seed-слоем.

### Model Synthesis / Merrell

Model synthesis — родственная семья алгоритмов constraint synthesis. Важный вывод для больших миров: можно генерировать чанками или проходами, снижая failure rate. Это особенно релевантно для карты 64x64, если тайлсет станет строгим.

Источник:
https://en.wikipedia.org/wiki/Model_synthesis

Ключевые выводы для проекта:

- для игровой карты лучше не пытаться решить весь мир одним хрупким супер-тайлсетом;
- можно делать macro constraints отдельно, а WFC применять к ограниченной семантической задаче;
- chunked / staged WFC может быть надёжнее, чем один монолитный solve.

### Oskar Stålberg / Bad North / Townscaper

По публичным описаниям и ссылкам из WFC-сообщества, Стольберг работал с 3D tiled model, 2D tiled model для irregular grids и интерактивными WFC/tileset-демо. Для Bad North особенно важен тезис про навигабельность: остров должен быть не просто красивым, а тактически и pathfinding-валидным. Townscaper показывает другой принцип: игрок задаёт намерение простыми блоками, а rule-based/procedural layer превращает конфигурацию в визуально богатую архитектуру.

Источники:
https://github.com/mxgmn/WaveFunctionCollapse
https://www.badnorth.com/
https://www.townscapergame.com/
https://oskarstalberg.com/game/wave/wave.html

Ключевые выводы для Fantasy Marka:

- не нужно отдавать WFC право решать всю топологию мира, если у нас уже есть игровые инварианты;
- WFC должен работать как интерпретатор/синтезатор поверх намерения генератора;
- как в Bad North, результат должен каждый шаг или каждый solve проходить gameplay validation: связность, доступность, стартовая зона, мосты/реки;
- как в Townscaper, ценно иметь irregular/organic результат, но он должен происходить из маленького набора понятных правил.

## 4. Главный риск

Полноценный WFC может ухудшить генератор, если дать ему слишком много власти.

Для Fantasy Marka уже есть сильные игровые инварианты:

- один остров;
- берег и море;
- горные хребты как entity;
- реки по границам гексов;
- стартовая зона с лугами и ресурсами;
- pathfinding, дороги, мосты, лес как блокирующий слой;
- headless smoke-test на 60 дней.

Если WFC начнёт решать воду, горы и реки одновременно, мы получим красивый, но хрупкий генератор. Ошибки будут проявляться не как визуальные дефекты, а как сломанная симуляция: нет места для ратуши, лес запер проходы, река отрезала карту, старт без дерева/камня.

## 5. Варианты внедрения

### Вариант A — заменить только `wgFill()` полноценным hex-cell WFC

WFC решает только `GRASS / FOREST / ROCK` на уже заданной карте, где `WATER` и `MTN` фиксированы. Реки, высота, берег и озёра становятся constraints/weights.

Плюсы:

- минимальный риск для симуляции;
- текущий рендер и dual-triangle transitions остаются почти без изменений;
- легко сравнить до/после;
- можно сделать полноценный WFC без переписывания всего worldgen.

Минусы:

- WFC не создаёт остров сам;
- эффект ближе к «биомный constraint solver», чем к генератору островов Bad North.

### Вариант B — WFC решает весь островный macro-layout

WFC получает тайлы вроде `sea`, `shore`, `plain`, `forest`, `hill`, `ridge`, `river`, `lake`, `cliff`, `start_basin` и сам собирает остров.

Плюсы:

- концептуально ближе к tile/model synthesis island generator;
- можно получать неожиданные формы.

Минусы:

- высокий риск contradiction и неиграбельных карт;
- потребуется большой тайлсет и глобальные constraints;
- реки по границам гексов сложнее вписать;
- текущие `wgIsland/wgRidges/genRivers` фактически придётся заменить.

### Вариант C — гибрид Oskar-style: macro skeleton + полноценный WFC-интерпретатор

Текущий generator продолжает задавать намерение: остров, вода, горы, реки, высота. Затем полноценный WFC получает preconstraints и решает биомно-топологический слой: луга, лесные массивы, скалы, предгорья, поймы, береговые уступы, просеки/коридоры.

Плюсы:

- WFC становится настоящим constraint solver;
- сохраняются игровые инварианты;
- можно добавить navigability-aware heuristic;
- эстетически ближе к Oskar-подходу: не шум, а rule-based синтез на irregular grid.

Минусы:

- это сложнее, чем просто переписать `wgFill()`;
- нужно аккуратно спроектировать tile semantics и compatibility.

Рекомендация research-этапа: **Вариант C**, но реализовывать его итерационно через Вариант A как первый milestone.

## 6. Предлагаемая рамка согласования

Перед проектированием нужно согласовать 5 решений.

### Решение 1. Область власти WFC

Рекомендация: WFC v1 не трогает `WATER` и `MTN`; они фиксированы. WFC решает только сушу между ними.

Альтернатива: WFC также может решать береговые типы, lake shore, cliffs. Это можно добавить во второй итерации.

### Решение 2. Гранулярность тайлов

Рекомендация: simple tiled model на hex-клетках. Совместимость по 6 направлениям `hexDirs(x)`.

Не рекомендую начинать с overlapping WFC по dual triangles: визуально заманчиво, но риск запутать gameplay-terrain и renderer-terrain.

### Решение 3. Тип тайлов

Рекомендация v1:

- `grass_open`
- `grass_riverplain`
- `forest_edge`
- `forest_core`
- `rock_foothill`
- `rock_lake_shore`
- `clear_corridor` / optional

Итоговый `S.terr` пока остаётся старым enum: `GRASS / FOREST / ROCK`; subtypes можно хранить отдельно в `S.wfcKind` или `S.biome`.

### Решение 4. Recovery strategy

Рекомендация: bounded backtracking + full restart на уровне `genWorld()` attempt.

- На маленькой карте 64x64 можно позволить несколько solve attempts.
- Backtracking нужен, чтобы это был полноценный WFC, но не стоит усложнять до бесконечного solver-а.

### Решение 5. Gameplay validation

Рекомендация: WFC solve считается успешным только если прошли проверки:

- `wgValidate()`;
- стартовая зона имеет достаточно лугов;
- лесные массивы не создают непроходимую пробку;
- есть коридоры к ключевым ресурсам;
- реки/мосты не ломают базовую связность;
- headless `node dist/game.js test-1` проходит.

## 7. Предварительные acceptance criteria

Для research/design-ветки успехом будет считаться не «красиво на одном сиде», а:

- старые сиды детерминированы;
- генератор не падает на серии сидов;
- нет silent fallback, который скрывает WFC-противоречия;
- debug replay показывает entropy/domain и propagation;
- текущий renderer продолжает работать через `S.terr`;
- новые WFC-subtypes не ломают pathfinding и экономику;
- smoke-test проходит в Node.

- тестовую матрицу сидов.

## 9. Research conclusion

Полноценный WFC для Fantasy Marka лучше делать не как замену всего генератора мира, а как constraint-solving интерпретатор поверх macro skeleton. Это ближе к полезной стороне Oskar Stålberg-подхода: маленький набор правил даёт органичный результат, но gameplay-инварианты остаются главнее эстетики.
