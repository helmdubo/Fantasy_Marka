# Дизайн: полноценный WFC-генератор карты (nested, два уровня)

Статус: **согласовано, ждёт реализации**. Решения владельца (2026-07-05):
двухуровневый (nested) WFC; расширение тайлсета переходными тайлами;
противоречия — бэктрекинг с лимитом глубины.

Референсы: Maxim Gumin (mxgmn/WaveFunctionCollapse), Boris the Brave
(DeBroglie, «WFC Explained», «Constraint-Based Tile Generators»), Paul
Merrell (Model Synthesis), Oskar Stålberg (Bad North, Townscaper, Planet).
Ссылки — в конце документа.

## 1. Итоги research (выжимка)

- WFC = CSP-солвер: домен возможных тайлов на клетку; цикл
  observe (клетка с минимальной энтропией Шеннона → взвешенный бросок) →
  propagate (арк-консистентность AC-4: стек удалений + счётчики
  «enabler'ов», O(1) на удаление). Пустой домен = противоречие.
- Гумин при противоречии рестартует; DeBroglie умеет бэктрекинг
  («мощно, но медленно — для небольших массивов»; 64×64 — небольшой).
- WFC сам не создаёт крупных структур («каша»): глобальную форму задают
  снаружи — pre-seed/pinning клеток, веса как функция позиции, зоны.
  Отсюда nested-схема: грубый уровень задаёт структуру, тонкий — детали.
- Столберг избегает противоречий дизайном тайлсета («no dead ends»:
  любое частичное решение достраиваемо) и держит универсальный тайл.
  У нас эту роль сохраняет GRASS — бэктрекинг остаётся страховкой для
  жёстких правил, а не основным механизмом.
- Проходимость Bad North: зашита в метаданные тайла — у нас то же через
  правило «FOREST граничит только с FOREST/SCRUB» (лес всегда в кольце
  проходимой опушки, коридоры by construction).

## 3. Движок WFC — новый модуль `src/02b_wfc.js`

Один переиспользуемый солвер для обоих уровней (и для пойм в п.6).
Обычный скрипт без import/export, строка в `MANIFEST` перед `02c`.

### API

```js
wfcSolve({
  n,                 // число клеток
  neighbors,         // (i) => [[j, dir], ...]  dir 0..D-1, D=4 или 6
  numTiles, opposite,// opposite[dir] — встречное направление
  adj,               // adj[t][dir] = битмаска допустимых соседей (Uint16)
  weights,           // (i) => Float32Array[numTiles], зависят от позиции
  pins,              // Map i -> tile (принудительные клетки)
  domains,           // опц.: стартовые битмаски (сужение доменов зон)
  rng,               // S.rng — детерминизм по сиду
  backtrackLimit,    // лимит откатов на прогон (макро 200, микро 800)
  trace,             // null | объект для дебаг-реплея
}) => Int8Array|null // тайл на клетку; null = лимит исчерпан (→ restart attempt)
```

### Структуры данных (AC-4, по gridbugs.org)

- `dom: Uint16Array(n)` — домен-битмаска (9 тайлов < 16).
- `enab: Uint8Array(n*T*D)` — счётчики enabler'ов: сколько тайлов в домене
  соседа по направлению d совместимы с тайлом t клетки i. 4096×9×6 ≈ 221 КБ.
- Кэш энтропии: `sumW`, `sumWLogW: Float32Array(n)`;
  `E = log(sumW) − sumWLogW/sumW` + крошечный шум `rng()*1e-4` для ничьих.
- Мин-куча по энтропии с ленивой инвалидацией (запись версии клетки).
- `removalStack: [cell, tile]` — очередь пропагации.

### Цикл

```
init: домены = все тайлы (или domains[i]); применить pins через ban+propagate
loop:
  cell = минимальная энтропия (пропуская схлопнутые); нет → успех
  tile = взвешенный бросок по оставшимся weights(cell)
  decide(cell, tile): checkpoint в undo-лог; ban всех прочих тайлов клетки
  propagate: снимать (c,t) со стека; для каждого соседа/направления
    декремент enab совместимых тайлов; счётчик 0 → ban → в стек
  противоречие (домен пуст) → backtrack
```

### Бэктрекинг с лимитом

- Undo-лог: плоский массив событий `(cell, tile)` всех ban'ов + маркеры
  решений (checkpoint = индекс в логе, клетка, выбранный тайл).
- Откат: пройти лог назад до маркера, вернуть биты домена и инкрементнуть
  счётчики enab; `sumW/sumWLogW` пересчитать по восстановленным доменам
  затронутых клеток (их список — те же события лога); перевставить в кучу.
- После отката: **ban выбранного тайла** в клетке решения (это ban без
  нового checkpoint'а — он принадлежит родительскому решению) → propagate.
  Домен опустел → откат ещё уровнем выше (каскад — дёшево, тот же лог).
- `backtrackLimit` исчерпан → `null` → внешний attempt-цикл рестартует
  генерацию с тем же `S.rng` (следующая попытка = другие броски).

### Кастом-ограничение «тонкие хребты» (микро-уровень)

Не парное правило — счётчик по окрестности, проверяется реактивно после
каждого схлопывания (как кастом-constraints DeBroglie):

- клетка схлопнулась в MTN → у каждого соседа, где MTN ещё возможен,
  посчитать схлопнутых MTN-соседей; их ≥ 2 → ban MTN у этого соседа;
- редкий тайл `MTN_NODE` (та же графика, отдельный id) допускает до 4 —
  из узлов растут отроги, форма «снежинки» сохраняется;
- ban'ы идут через общий undo-лог — бэктрекинг отменяет их автоматически.

## 4. Макро-уровень: регионы 16×16

Суперклетка 4×4 гекса; 4-соседство (N/S/E/W); `opposite` тривиален.
Квадратная сетка сознательно: регионы — карта весов, а не геометрия;
органику швов даёт интерполяция весов и микро-WFC.

Тайлы регионов (8): `R_SEA, R_COAST, R_PLAIN, R_FOREST, R_WETLAND,
R_HILLS, R_PEAK, R_LAKE`.

Матрица соседств (симметричная; главное):

| ↔          | SEA | COAST | PLAIN | FOREST | WETLAND | HILLS | PEAK | LAKE |
|------------|-----|-------|-------|--------|---------|-------|------|------|
| SEA        |  +  |   +   |   —   |   —    |    —    |   —   |  —   |  —   |
| COAST      |  +  |   +   |   +   |   +    |    +    |   +   |  —   |  —   |
| PLAIN      |  —  |   +   |   +   |   +    |    +    |   +   |  —   |  +   |
| FOREST     |  —  |   +   |   +   |   +    |    +    |   +   |  —   |  +   |
| WETLAND    |  —  |   +   |   +   |   +    |    +    |   —   |  —   |  +   |
| HILLS      |  —  |   +   |   +   |   +    |    —    |   +   |  +   |  —   |
| PEAK       |  —  |   —   |   —   |   —    |    —    |   +   |  +   |  —   |
| LAKE       |  —  |   —   |   +   |   +    |    +    |   —   |  —   |  —   |

By construction: море отделено от суши поясом COAST; PEAK всегда в кольце
HILLS (предгорья); озёра не касаются моря и берега (средний пояс, как
`wgLakes` сейчас); болота у озёр/низин, не в предгорьях.

- **Pins**: весь периметр сетки = `R_SEA` (border-constraint по DeBroglie).
- **Веса от позиции**: `w(R_SEA) = f(d)` растёт к краю, веса суши растут к
  центру (радиальный градиент + fbm — форма острова); `w(R_PEAK)`
  максимален в глубине.
- **Счётные инварианты** — не в солвере, а в валидаторе макро-результата
  (дешевле честных count-constraints): 1–3 PEAK-кластера, 1–2 LAKE,
  суша связна (flood-fill), доля суши 30–72%. Провал → пересбор макро
  (сотни микросекунд на 256 клеток, можно позволить до ~20 попыток).

## 5. Микро-уровень: террейн 64×64

### Тайлсет (9): расширение `T` в `00_config.js`

| id | тайл    | роль                                   | проходимость |
|----|---------|----------------------------------------|--------------|
| 0  | WATER   | море/озеро/пруд (waterKind как сейчас) | нет          |
| 1  | GRASS   | луг, универсальный тайл                | все          |
| 2  | FOREST  | лес-ядро, ресурс дерева                | герои        |
| 3  | ROCK    | скалы/холмы, бедные шахты              | нет (как сейчас) |
| 4  | MTN     | хребет, шахты                          | нет          |
| 5  | SAND    | пляж (новый)                           | все          |
| 6  | SCRUB   | опушка/кустарник (новый)               | все          |
| 7  | SWAMP   | болото (новый)                         | все, строить нельзя |
| 8  | MTN_NODE| узел хребта (новый id, графика MTN)    | нет          |

### Матрица соседств (жёсткая)

| ↔      | WATER | GRASS | FOREST | ROCK | MTN* | SAND | SCRUB | SWAMP |
|--------|-------|-------|--------|------|------|------|-------|-------|
| WATER  |   +   |   +   |   —    |  +   |  —   |  +   |   —   |   +   |
| GRASS  |   +   |   +   |   —    |  +   |  —   |  +   |   +   |   +   |
| FOREST |   —   |   —   |   +    |  —   |  —   |  —   |   +   |   —   |
| ROCK   |   +   |   +   |   —    |  +   |  +   |  —   |   +   |   —   |
| MTN*   |   —   |   —   |   —    |  +   |  +   |  —   |   —   |   —   |
| SAND   |   +   |   +   |   —    |  —   |  —   |  +   |   +   |   —   |
| SCRUB  |   —   |   +   |   +    |  +   |  —   |  +   |   +   |   +   |
| SWAMP  |   +   |   +   |   —    |  —   |  —   |  —   |   +   |   +   |

MTN* = MTN и MTN_NODE (одинаковые правила смежности + кастом-счётчик
«тонкости» из п.3). Ключевые следствия by construction:

- **лес всегда в кольце SCRUB** → у жителей гарантированные обходы,
  лесоруб рубит с опушки, «крапинок» леса не бывает (замена релаксации);
- **MTN только через ROCK** → предгорья вокруг каждого хребта;
- лес и горы никогда не касаются воды; берег — GRASS/SAND/ROCK (утёсы)/
  SWAMP (топкое устье);
- GRASS совместим со всем, кроме FOREST/MTN — почти-универсальный тайл:
  противоречия редки, бэктрекинг короткий.

### Веса

`weights(i)` = произведение:
1. **приор региона** (суперклетки клетки i), билинейно интерполированный
   между 4 ближайшими суперклетками — швы регионов размываются;
   примеры приоров: R_FOREST → {FOREST 3.0, SCRUB 1.2, GRASS 0.7};
   R_HILLS → {ROCK 2.5, GRASS 1.0, SCRUB 0.5, MTN 0.35, MTN_NODE 0.04};
   R_COAST → {GRASS 1.6, SAND 1.2, ROCK 0.5, SCRUB 0.4};
   R_WETLAND → {SWAMP 1.8, GRASS 1.2, SCRUB 0.6, WATER 0.25};
   R_SEA → {WATER 30, SAND 0.4, ROCK 0.2, GRASS 0.15} (берег органичный);
2. **fbm-шум** per-тайл (лесной шум, каменный шум — разные октавы/сдвиги),
   чтобы внутри региона были кластеры, а не равномерная каша;
3. аффинность больше не нужна — её работу выполняют жёсткие правила.

- **Домены-зоны**: суперклетки R_SEA, не граничащие с сушей, стартуют с
  доменом {WATER} — меньше работы солверу.
- **Pins**: центр каждого PEAK-региона → MTN_NODE (гарантия хребта);
  центр LAKE-региона → WATER.

### Порядок и производительность

4096 клеток × 9 тайлов × 6 направлений, AC-4 — оценочно 10–40 мс в JS на
телефоне, плюс бэктрекинг (лимит 800 откатов). Худший случай с 5 attempt —
доли секунды; текущий генератор уже живёт в этом бюджете.

## 6. Реки и поймы

- `S.elev`: базовые высоты регионов (SEA 0, COAST .12, WETLAND .2,
  PLAIN .3, FOREST .4, LAKE .25, HILLS .65, PEAK 1.0), билинейная
  интерполяция на гексы + fbm ±0.08; после микро-WFC поклеточное
  уточнение: WATER→0, MTN→max(elev,.8), ROCK→+0.1. Монотонность к морю
  обеспечена радиальной структурой макро (COAST-пояс) — реки текут вниз,
  как сейчас.
- `genRivers` — без изменений (работает по `S.elev` и фактической воде).
- **Поймы (опционально, этап 2)**: инкрементальный пересбор по Столбергу —
  клетки, прилегающие к речным рёбрам, помечаются «грязными», их домен
  пересчитывается с новыми весами (GRASS/SWAMP ↑, ROCK ↓) и запретом
  FOREST у самой воды; пересобираются только выпавшие из домена клетки,
  пропагация локальна. Это тот же `wfcSolve` с `domains` = текущие тайлы
  везде, кроме грязных. До реализации этапа 2 поведение прежнее: река
  проходит сквозь любой террейн («поймы мягкие»).

## 7. Entity и валидация

- `ranges`: floodfill по MTN|MTN_NODE + классификация узлов по степени
  (существующий код из `wgRidges` переезжает почти как есть).
- `forests`: floodfill по FOREST (SCRUB — не лес, в entity не входит).
- `lakes`: как сейчас через `classifyWater`/`waterComps`.
- `wgValidate` расширить: доли новых тайлов (SWAMP ≤ 8% суши, SAND ≤ 6%),
  «каждый PEAK-регион дал хребет ≥ 4 клеток», связность суши по `S.pass`
  (жительская!) от берега — гарантия, что кольца SCRUB действительно
  дали коридоры.

## 8. Геймплей новых тайлов (минимальный скоуп)

- **SCRUB**: проходим всем, строить можно; лесоруб рубит смежный FOREST
  (как с GRASS); регенерация леса из пеньков — без изменений; в
  `thSiteScore` считается как луг с лёгким штрафом.
- **SAND**: проходим, строить нельзя (кроме причала/порта — они и так
  требуют берег); ресурсов нет.
- **SWAMP**: проходим всеми, строить нельзя; задел на будущее
  (знахарские травы). Скорость не штрафуем (в движке нет пер-тайловой
  цены шага — не заводим ради этого).
- **MTN_NODE**: во всей игровой логике = MTN (один `isMtn(t)`-хелпер);
  отдельный id нужен только солверу и, позже, — квестам («сердце хребта»).
- Рендер/атлас: временные перекраски (SAND — светлый песочный, SCRUB —
  жёлто-зелёный с редкими кустами, SWAMP — тёмный сине-зелёный);
  полноценные спрайты — в общей перерисовке под новый масштаб.

## 9. Дебаг-реплей

`wfcReplay` сохраняется и расширяется. Трейс `S.wfcTrace`:
`{macro:[{i,t}...], pins:[], steps:[{i,t}...], bans опц., backtracks:[{from,to}]}`.
Реплей: сначала макро-сетка крупными клетками, затем волна микро-уровня в
фактическом порядке схлопывания (порядок теперь энтропийный — «расползание
пятнами» вместо BFS-колец, это само по себе наглядно), **красные вспышки =
откаты бэктрекинга** (вместо нынешней релаксации), в конце — рёбра рек.

## 11. Риски

- **Форма острова** из макро-WFC менее предсказуема, чем радиальный fbm:
  страхуют радиальные веса + макро-валидатор с дешёвыми ретраями.
- **Ступенчатость elevation** по регионам → странные реки: лечится
  интерполяцией + шумом; проверять на эталонных сидах.
- **Баланс площадей**: кольцо SCRUB съедает часть леса; доля FOREST
  регулируется весами и порогом в `wgValidate` (уже есть «мало леса»).
- **Бэктрекинг + жёсткая матрица**: если лимиты будут выбиваться —
  сначала ослаблять матрицу (разрешить SWAMP↔ROCK и т.п.), а не наращивать
  лимит: почти-универсальный GRASS должен делать откаты редкими.
- **Экономика с непроходимым лесом** уже отмечена в HANDOFF как «крутить
  по фидбеку» — новые коридоры SCRUB, скорее всего, её улучшат, но темп
  добычи дерева проверить headless-прогоном.
