# Дизайн v2: WFC-генератор карты (вороной-патчи → высоты → террейн)

Статус: **согласовано, зафиксировано, ждёт реализации**.

Решения владельца (2026-07-05, раунд 1): переходные тайлы; бэктрекинг с
лимитом глубины; nested-подход (несколько слоёв WFC).
Решения раунда 2 (по примерам `docs/Examples_procgen`): отойти от
островного принципа — **континент с морским краем**; крупную структуру
задают **вороной-слой патчей** и **слой дискретных высот** (solver в духе
Loren Schmidt) вместо квадратной макро-сетки 16×16 из v1; уровни высоты —
**служебные** (рендер не меняется).

Референсы: Maxim Gumin (mxgmn/WaveFunctionCollapse), Boris the Brave
(DeBroglie), Paul Merrell (Model Synthesis), Oskar Stålberg (Bad North,
Townscaper, Planet), Loren Schmidt (elevation/terrain solver), Samuel
Lundsten (WFC-навигабельность, граф-карты). Ссылки — в конце.

## 1. Ключевые идеи (из research + примеров владельца)

- WFC = CSP: домен возможностей на клетку, матрица допустимых соседств,
  выбор клетки с минимальной энтропией Шеннона, AC-4-пропагация
  (счётчики enabler'ов, O(1) на удаление), бэктрекинг при противоречии.
- Крупные структуры из локальных правил возможны двумя путями:
  **упорядоченный домен** (домен = уровень высоты, правило |Δh|≤1 —
  гора обязана иметь широкое основание, хребты эмерджентны; это
  slope-constraint-солвер Loren Schmidt) и **дизайн тайлсета**
  (кольца опушки вокруг леса, навигабельность Bad North).
  Но структурный слой перед солвером всё равно нужен: у Шмидт это
  вороной-патчи с параметрами («some parts plains, some mountains»),
  у Лундстена — граф POI до «одевания» тайлами.
- Стратегия Столберга против противоречий: почти-универсальный тайл
  (у нас GRASS) + полнота тайлсета; бэктрекинг — страховка, не основа.
- Сетка: WFC решаем **по гексам** (вся симуляция гексовая; 6 соседей
  выразительнее 3 у треугольника, клеток вдвое меньше). Треугольники
  dual-сетки остаются для рёберных рек и переходных тайлов рендера.
  Corner-based тайлы по вершинам гексов (Townscaper) — возможный
  будущий рендер-слой, не симуляция.

## 2. Целевой пайплайн `genWorld`

```
attempt-цикл (до 5, как сейчас):
  1. ВОРОНОЙ      Poisson-диск сайты (~28-40) -> патчи по гексам
                  (multi-source BFS) -> граф смежности патчей
  2. РОЛИ ПАТЧЕЙ  WFC №1 на графе патчей (нерегулярная «сетка»):
                  SEA / LOWLAND / WETLAND / UPLAND / HIGHLAND / LAKE
                  континент: морской край по выбранному азимуту
  3. ВЫСОТЫ       WFC №2 на гексах: домен = уровни 0..6, |дельта h|<=1,
                  веса тянут к диапазону роли патча, море pin 0
  4. ТЕРРЕЙН      WFC №3 на гексах: 9 тайлов, домены сужены высотой,
                  жёсткая матрица соседств, веса от патча и шума
  5. ВОДА/ПОЛЯ    classifyWater, S.distCoast, S.elev = level/6 + шум
  6. РЕКИ         genRivers v2 без изменений (вниз по высотам, рёберные)
  7. ENTITY       ranges/forests/lakes floodfill-ом (адаптация wgEntities)
  8. ВАЛИДАЦИЯ    wgValidate v2; ошибка -> следующий attempt
```

Один движок WFC прогоняется трижды — на графе патчей и дважды на гексах.
Удаляются: `wgIsland`, `wgLakes`, `wgRidges`, `wgElevation`, `wgFill`,
релаксация. Остаются: `genRivers`, `classifyWater`, `wgBfsField`,
`wgEntities` (адаптируется), `wgValidate` (расширяется), attempt-цикл.

## 3. Движок WFC — новый модуль `src/02b_wfc.js`

Обычный скрипт без import/export, строка в `MANIFEST` перед `02c`.
Сетко-независимый: клетки — индексы, соседство — колбэк.

### API

```js
wfcSolve({
  n,                 // число клеток (патчей или гексов)
  neighbors,         // (i) => [[j, dir], ...]; для графа патчей dir=0
  numTiles, opposite,// opposite[dir]; у графа патчей — симметрия dir 0
  adj,               // adj[t][dir] = битмаска допустимых соседей (Uint16)
  weights,           // (i) => Float32Array[numTiles] — зависят от позиции
  pins,              // Map i -> tile (принудительные клетки)
  domains,           // опц.: стартовые битмаски (сужение по слою выше)
  rng,               // S.rng — детерминизм по сиду
  backtrackLimit,    // патчи 100 / высоты 600 / террейн 800
  onCollapse,        // опц. колбэк кастом-ограничений (тонкие хребты)
  trace,             // null | объект для дебаг-реплея
}) => Int8Array|null // тайл на клетку; null = лимит исчерпан -> restart
```

### Структуры данных (AC-4, по gridbugs.org)

- `dom: Uint16Array(n)` — домен-битмаска (макс. 9 тайлов/7 уровней < 16).
- `enab: Uint8Array(n*T*D)` — счётчики enabler'ов (гексы: 4096×9×6 ≈ 221 КБ).
- Кэш энтропии: `sumW`, `sumWLogW: Float32Array(n)`;
  `E = log(sumW) − sumWLogW/sumW` + шум `rng()*1e-4` для ничьих.
- Мин-куча по энтропии с ленивой инвалидацией.
- `removalStack: [cell, tile]` — очередь пропагации.

### Цикл

```
init: домены = все тайлы (или domains[i]); pins через ban+propagate
loop:
  cell = минимальная энтропия среди несхлопнутых; нет -> успех
  tile = взвешенный бросок по оставшимся weights(cell)
  decide(cell, tile): checkpoint в undo-лог; ban прочих тайлов клетки
  propagate: снимать (c,t) со стека; соседям декремент enab
    совместимых тайлов; счётчик 0 -> ban -> в стек; onCollapse-баны туда же
  домен пуст -> backtrack
```

### Бэктрекинг с лимитом

- Undo-лог: плоский массив ban-событий `(cell, tile)` + маркеры решений
  (checkpoint = индекс лога, клетка, выбранный тайл).
- Откат: пройти лог до маркера, вернуть биты домена, инкрементнуть enab,
  пересчитать sumW/sumWLogW затронутых клеток, перевставить в кучу.
- После отката ban выбранного тайла (принадлежит родительскому решению)
  → propagate; домен пуст → каскадный откат выше.
- Лимит исчерпан → `null` → attempt-цикл рестартует генерацию.

## 4. Слой 1: вороной-патчи и их роли

### Разбиение

- Poisson-диск сэмплинг сайтов на карте 64×64: минимальная дистанция
  ~7-9 гексов → 28-40 сайтов (через дartboard-отбор на `S.rng`).
- Патчи: multi-source BFS от сайтов по гексам (`wgBfsField`-подход) —
  каждый гекс получает id ближайшего сайта; границы органично рваные
  (BFS по гексам, не евклид). Хранится `S.patch: Int16Array(N)`.
- Граф патчей: пары патчей, имеющие соседние гексы. Это нерегулярная
  сетка а-ля Stålberg Planet — по ней идёт WFC №1 (dir один, симметрия).

### Роли (6) и матрица смежности графа

| ↔        | SEA | LOWLAND | WETLAND | UPLAND | HIGHLAND | LAKE |
|----------|-----|---------|---------|--------|----------|------|
| SEA      |  +  |    +    |    +    |   +    |    —     |  —   |
| LOWLAND  |  +  |    +    |    +    |   +    |    —     |  +   |
| WETLAND  |  +  |    +    |    +    |   +    |    —     |  +   |
| UPLAND   |  +  |    +    |    +    |   +    |    +     |  +   |
| HIGHLAND |  —  |    —    |    —    |   +    |    +     |  —   |
| LAKE     |  —  |    +    |    +    |   +    |    —     |  —   |

HIGHLAND только через UPLAND (предгорья by construction); озёра не у
моря и не в горах; SEA↔UPLAND разрешено — обрывистые берега-утёсы.

### Континент с морским краем (замена острова)

- Выбирается случайный «морской азимут» θ (вектор от центра карты).
- Патчи, касающиеся границы карты: скалярное произведение направления
  на патч с θ > порога → **pin SEA**; остальные краевые → домен без SEA
  (суша «уходит за карту» — неизведанная глубь материка, лор Марки).
- Веса: SEA дотягивается вглубь весом (море ~15-35% площади),
  HIGHLAND тяготеет к дальней от моря части (вес ∝ дистанции до
  морского края по графу патчей).
- Счётные инварианты — валидатором слоя (дёшево, до ~20 пересборов):
  доля моря 12-35%, 1-3 HIGHLAND-кластера, 0-2 LAKE, суша связна,
  морской край непрерывен вдоль своего участка границы.

## 5. Слой 2: дискретные высоты (solver в духе Loren Schmidt)

- Домен клетки: уровни 0..6 (упорядоченный!). Море (гексы SEA-патчей)
  pin 0; LAKE-патчи pin 1. Суша: домен 1..6.
- Единственное жёсткое правило: **|h(a) − h(b)| ≤ 1** для соседей
  (матрица трёхдиагональная: уровень k совместим с k-1,k,k+1).
  Из него эмерджентно: горы имеют широкие основания, хребты и долины —
  крупные структуры из локального правила.
- Веса тянут к целевому диапазону роли патча (домены НЕ режем жёстко —
  на границах патчей склоны строит пропагация):
  LOWLAND→{1,2}, WETLAND→{1}, UPLAND→{2..4}, HIGHLAND→{4..6};
  внутри диапазона лёгкий fbm-шум для рельефности.
- Высоты служебные: `S.elevL: Int8Array` (уровни) и
  `S.elev = level/6 + fbm*0.04` (float для genRivers). Рендер не меняем.
- Реки текут «строго вниз по уровням» — надёжнее нынешнего поля;
  водопады (`r_falls`) естественно ложатся на перепады уровней.

## 6. Слой 3: террейн 64×64

### Тайлсет (9): расширение `T` в `00_config.js`

| id | тайл    | роль                                   | проходимость |
|----|---------|----------------------------------------|--------------|
| 0  | WATER   | море/озеро/пруд (waterKind как сейчас) | нет          |
| 1  | GRASS   | луг, почти-универсальный тайл          | все          |
| 2  | FOREST  | лес-ядро, ресурс дерева                | герои        |
| 3  | ROCK    | скалы/холмы, бедные шахты              | нет (как сейчас) |
| 4  | MTN     | хребет, шахты                          | нет          |
| 5  | SAND    | пляж (новый)                           | все          |
| 6  | SCRUB   | опушка/кустарник (новый)               | все          |
| 7  | SWAMP   | болото (новый)                         | все, строить нельзя |
| 8  | MTN_NODE| узел хребта (новый id, графика MTN)    | нет          |

### Связка с высотами (сужение доменов до старта)

- h=0 → {WATER}; h=1 → без MTN/MTN_NODE/ROCK; h=2..3 → без WATER/MTN;
  h=4 → + ROCK; h≥5 → {MTN, MTN_NODE, ROCK, GRASS} (альпийские луга).
- SAND только в клетках, смежных с морем (h=1 у кромки).
  SWAMP только h=1. Домены сужаются заранее — солверу меньше работы.

### Матрица соседств (жёсткая)

| ↔      | WATER | GRASS | FOREST | ROCK | MTN* | SAND | SCRUB | SWAMP |
|--------|-------|-------|--------|------|------|------|-------|-------|
| WATER  |   +   |   +   |   —    |  +   |  —   |  +   |   —   |   +   |
| GRASS  |   +   |   +   |   —    |  +   |  —   |  +   |   +   |   +   |
| FOREST |   —   |   —   |   +    |  —   |  —   |  —   |   +   |   —   |
| ROCK   |   +   |   +   |   —    |  +   |  +   |  —   |   +   |   —   |
| MTN*   |   —   |   —   |   —    |  +   |  +   |  —   |   —   |   —   |
| SAND   |   +   |   +   |   —    |  —   |  —   |  +   |   +   |   —   |
| SCRUB  |   —   |   +   |   +    |  +   |  —   |  +   |   +   |   +   |
| SWAMP  |   +   |   +   |   —    |  —   |  —   |  —   |   +   |   +   |

MTN* = MTN и MTN_NODE. Следствия by construction: **лес всегда в кольце
SCRUB** (коридоры жителям, лесоруб рубит с опушки, замена релаксации);
MTN только через ROCK; лес и горы не касаются воды; берег — GRASS/SAND/
ROCK-утёс/SWAMP-топь. GRASS совместим со всем, кроме FOREST/MTN —
противоречия редки, бэктрекинг короткий.

### Кастом-ограничение «тонкие хребты» (onCollapse)

- Клетка схлопнулась в MTN → соседям, где MTN ещё возможен, посчитать
  схлопнутых MTN-соседей; ≥2 → ban MTN (через общий undo-лог).
- `MTN_NODE` (редкий вес) допускает до 4 — узлы и отроги «снежинки».
- Хребты возникают цепями вдоль гребней h=5-6 (высотный слой уже
  проложил «русла» для них) — структура двойная: эмерджентная + правило.

### Веса

`weights(i)` = приор роли патча (интерполяция к соседним патчам у границ)
× fbm-шум per-тайл (лесной/каменный — разные октавы) × высотный акцент
(ROCK тяжелее на перепадах уровней — обрывы каменистые). Аффинность
соседей больше не нужна: кластеры дают жёсткие правила и шум.

## 7. Реки, entity, валидация

- `genRivers` — без изменений: `S.elev` из уровней, вода из террейна.
- Поймы (этап 2, после одобрения картинки): «editable WFC» по
  Столбергу — клетки у речных рёбер получают новые веса (GRASS/SWAMP ↑),
  пересборка только «грязных» клеток тем же `wfcSolve` с `domains`.
- `ranges`: floodfill MTN|MTN_NODE + классификация узлов по степени
  (код из `wgRidges` переезжает). `forests`: floodfill FOREST (SCRUB не
  входит). `lakes`: LAKE-патчи + `classifyWater` как сейчас.
- `wgValidate` v2: доля моря 12-35%, суша связна по `S.pass` (жительская
  проходимость! гарантия коридоров), хребет ≥ 4 клеток на HIGHLAND,
  лес 8-42% суши, SWAMP ≤ 8%, SAND ≤ 6%, реки есть, порт возможен
  (есть морской берег с GRASS/SAND рядом).

## 8. Геймплей новых тайлов (минимальный скоуп)

- **SCRUB**: проходим всем, строить можно; лесоруб рубит смежный FOREST;
  в `thSiteScore` — как луг с лёгким штрафом.
- **SAND**: проходим, строить нельзя (кроме причала/порта); ресурсов нет.
- **SWAMP**: проходим всеми, строить нельзя; задел на знахарские травы.
  Штраф скорости не вводим (нет пер-тайловой цены шага — не заводим).
- **MTN_NODE**: во всей логике = MTN (хелпер `isMtn(t)`); отдельный id —
  для солвера и будущих квестов («сердце хребта»).
- Рендер/атлас: временные перекраски (SAND светло-песочный, SCRUB
  жёлто-зелёный с кустами, SWAMP тёмный сине-зелёный); полноценные
  спрайты — в общей перерисовке под новый масштаб.

## 9. Дебаг-реплей

`wfcReplay` расширяется до стадий (в духе видео Шмидт):
1) цветные вороной-патчи с ролями; 2) волна высот (оттенки серого);
3) волна террейна в порядке энтропийного схлопывания; красные вспышки =
откаты бэктрекинга; 4) рёбра рек. Трейс `S.wfcTrace`:
`{patches, roles, elevSteps, steps, backtracks, pins}`.

## 10. Порядок реализации (следующая сессия)

1. `src/02b_wfc.js`: движок (AC-4, куча, undo-лог, бэктрекинг, onCollapse)

2. Вороной-слой + WFC ролей на графе патчей; валидатор слоя;
   визуализация патчей в реплее.

   на сидах test-1 / world-7 / rnd-42.
4. Террейн-слой: тайлсет 9, сужение доменов высотой, матрица, веса,
   «тонкие хребты».
5. Новые тайлы в геймплей: `T`, `S.pass`/`S.passHero`, стройка,
   `thSiteScore`, атлас-перекраски.
6. Entity + `wgValidate` v2; адаптация `wgEntities`.
7. Реплей-дебаг; `node dist/game.js test-1 [quest]` + сиды.
8. Балансировка весов по скриншотам (Playwright + локальный three.min.js).
9. Этап 2 (отдельно): поймы инкрементальным пересбором; бэклог — граф

## 11. Риски

- **Три слоя = три источника фейлов**: каждый слой валидируется и
  пересобирается независимо (патчи дёшево, высоты/террейн — по
  backtrackLimit); attempt-цикл — последняя страховка.
- **Склоны съедают место**: |Δh|≤1 между морем (0) и HIGHLAND (6)
  требует ≥6 клеток градиента — HIGHLAND не должен подходить к морю
  ближе ~8 клеток; правило SEA↮HIGHLAND в графе патчей это уже даёт
  (патч ≥ 7 гексов диаметром), но проверить на сидах.
- **Морская торговля**: море теперь только с одного края — путь корабля
  и позиция порта могут удлиниться; проверить квест-прогон headless.
- **Ступенчатость рек**: плато одного уровня — реки могут «растекаться»;
  в `S.elev` добавлен fbm-джиттер для тай-брейка, как и сейчас.
- **Баланс площадей**: кольца SCRUB съедают лес; крутить весами и
  порогами `wgValidate` («мало леса» уже есть).
- **Бэктрекинг выбивает лимиты** → сначала ослаблять матрицу (например,
  SWAMP↔ROCK), а не наращивать лимит: GRASS должен делать откаты редкими.
