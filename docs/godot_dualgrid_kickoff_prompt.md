## Базовая геометрия (переносится как есть)

- **Гексы — слой данных.** Axial-координаты, flat-top, прямоугольная выкладка:
  `rMin(q) = -floor(q/2)`, ряд `i = r - rMin(q)`. Соседи:
  `DIRS = [(1,0),(1,-1),(0,-1),(-1,0),(-1,1),(0,1)]`.
- **Треугольники — дуальная сетка рендера**: вершины треугольников = центры гексов.
  На каждый гекс `(q,r)` два треугольника: up `{(q,r),(q+1,r),(q,r+1)}`,
  down `{(q+1,r),(q,r+1),(q+1,r+1)}`.
- Терминология Амита: центры гексов = его corners; центры треугольников = узлы
  corner-графа. Реки (позже) пойдут по рёбрам гексов = через центры треугольников;
  дороги — по рёбрам треугольников (гекс-граф). Граф рек тривалентен.

## Решение по топологии (зафиксировано, не пересматривать)

- Топология — **регулярный hex dual grid**, как описано выше. НЕ нерегулярная
  сетка Townscaper-типа (merge квадов / subdivide / relax): город будет строиться
  и расширяться по структурам сетки, поэтому слой данных обязан оставаться
  предсказуемым — адресация `(q,r)`, соседи по `DIRS`, ровно два треугольника
  на внутреннее ребро.
- «Убрать тайловый паттерн» — задача только слоя рендера: **детерминированный
  джиттер вершин** (чистая функция хеша `(q,r)` и сида, амплитуда ≤ 0.25 длины
  ребра) + 1–2 итерации Laplacian-релаксации визуальных позиций. Логика не
  меняется: соседство, пикинг, размещение построек считаются по каноническим
  центрам; визуальные позиции — отдельный массив `visualPos[]`, вычисляемый
  из канонических. Топология (индексы треугольников) идентична регулярной.
- Инварианты при этом сохраняются автоматически: марчинг по сигнатурам углов
  (INV-T1) и 1-Lipschitz (INV-T2) — свойства топологии, не метрики; джиттер
  детерминирован сидом (INV-T4/T5).
- В ядре доступ к соседству — через методы топологии (`NeighborsOf`,
  `TrianglesOfHex`, `HexesOfTriangle`), а не рассыпанную по коду арифметику:
  если позже захочется иная топология, замена останется локальной.

## Трёхслойная архитектура и инварианты (соблюдать строго)

1. **Слой 1 — данные (гексы)**: авторитетный. Тип/уровень поверхности на гекс.
2. **Слой 2 — маски/меш (треугольники)**: чисто детерминированный марчинг по
   битности углов. Ничего не решает, только читает слой 1.
3. **Слой 3 — кластеры/детекторы**: производные entity над слоем 1.

- **INV-T1 (Wang-контракт)**: треугольник определяется исключительно сигнатурой
  своих трёх углов; контекст влияет только на вариант интерьера, не на границу.
- **INV-T2 (1-Lipschitz)**: соседние узлы высотного поля отличаются ≤ 1 уровня;
  рендер многоуровневости = stacked passes одного и того же набора масок.
- **INV-T4**: вариативность — детерминированный хеш позиции, пул контрактов не растёт.
- **INV-T5**: вся кластеризация/генерация детерминирована сидом (реплеи,
  headless-валидация).

## Что именно построить в этой сессии (майлстоуны)

**M1 — сетка и меш.** C#-ядро: HexGrid (axial, прямоугольная выкладка W×H≈32×32),
маппинг гекс↔мир для flat-top; генерация дуальной треугольной сетки. Рендер:
один `ArrayMesh`/`MeshInstance3D` (перестраиваемый), плоская суша/вода по типу
гекса, камера-орбита, свет. Плюс переключаемый «органический» режим (см. раздел
про топологию): джиттер+релаксация `visualPos[]`, тогл на клавишу — с ним и без
него меш обязан оставаться водонепроницаемым, потому что топология одна и та же.
Проверка: сетка без дыр и швов, нормали корректны.

**M2 — марчинг переходов в 3D.** Типы гексов: вода < земля < предгорье < гора
(уровни расширяемы). Высота ВЕРШИНЫ треугольника = уровень её гекса (это и есть
3D-эквивалент 2D-марчинга: непрерывность гарантирована конструктивно — соседние
треугольники разделяют вершины). Скос между уровнями — интерполяция по
треугольнику; материал/цвет — **по крутизне склона, не по типу клетки** (урок
Оскара): крутое = скала, пологое = трава, сетка «растворяется» на склонах.
Вода — отдельная плоскость с прозрачностью.

**M3 — рельеф из кластеров.** Перенос из 02d_relief.js: детерминированная жадная
кластеризация горных гексов (розетки: гекс+6 соседей → пик; тройки: 3 взаимно
смежных; остаток: цепи/кольца/массивы/одиночки; эксклюзивность, жадный порядок,
tie-break по минимальному ключу, lookahead «не клеймить тройку, если блокирует
розетку 6/7»). Псевдовысоты: ранги-источники (пик 5.0, кольцо 4.0, узел 3.2,
хребет 2.6, массив 2.4, одиночка 1.8) → chamfer-релаксация `E=max(E, maxNb−1)`
→ поле автоматически 1-Lipschitz. В 3D E — это буквально высота вершин (а не
пиксельные пояса): гора «вырастает» из кластеров. Снежная шапка — материал по
порогу E≥4.5 (пробивает только центр розетки). Crease-линии хребтов: жёсткие
нормали (без сглаживания) на рёбрах с изломом выше порога.

**M4 — интерактив.** Тап/клик по гексу циклически меняет тип (вода→земля→гора),
меш перестраивается инкрементально (только затронутый чанк), кластеры
передетектируются событийно (dirty-окрестность радиуса 1-2). Это режим
«скульптинга» как в Townscaper и как в моём 2D-прототипе. Пикинг — raycast в
меш → индекс треугольника → логический гекс через топологию (не через обратную
формулу от мировых координат): так клик корректен и в органическом режиме.

**M5 — инфраструктура воспроизводимости.** Headless smoke-тест: сцена/скрипт,
запускаемый `godot --headless`, генерит N случайных карт по сидам, гоняет
кластеризацию и проверяет инварианты (1-Lipschitz, эксклюзивность кластеров,
детерминизм: один сид → одинаковая сигнатура кластеров), выход с ненулевым
кодом при нарушении. Плюс gdUnit4 или простые ассерты — на твой выбор, но
запуск из CLI обязателен.

НЕ в скоупе этой сессии (следующие): реки (priority flood по центрам
треугольников), биомы/влажность, knowledge-слой (туман), декор.

## Технические требования

- **Godot 4.4 .NET** (зафиксируй точную версию в README и не смешивай API
  Godot 3.x — частая ошибка LLM; при сомнении сверяйся с установленной версией).
- **Ядро симуляции/генерации — чистый C#** без зависимостей от Godot
  (отдельная папка/namespace, плоские массивы, сид mulberry32 или
  System.Random с фиксированным сидом): оно должно тестироваться и без движка.
  Godot-слой — тонкий: строит ArrayMesh из данных ядра.
- Сцены/ресурсы — текстовые (.tscn/.tres), всё в git. Коммить осмысленными
  шагами после каждого майлстоуна, с работающим состоянием.
- Заведи **CLAUDE.md** с конвенциями проекта: версия Godot, структура папок,
  команда headless-теста, правило «ядро не трогает Godot API», якорь на этот
  промт как источник инвариантов.
- Перф-бюджет: полная перестройка меша 32×32 < 16 мс; инкрементальная — чанк.
